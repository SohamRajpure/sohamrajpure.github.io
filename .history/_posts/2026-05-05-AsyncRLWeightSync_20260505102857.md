I've been working on an RFC ([#63120](https://github.com/ray-project/ray/issues/63120)) to add a Ray-native weight update API for SGLang servers. This post is a less formal writeup of the problem, the design we landed on, and the open questions I'm still thinking through.

{:toc}

## The Problem

In multi-turn RL — RLHF, online DPO, any setup where the model generates rollouts that feed into training — you're running two things concurrently: a trainer updating model weights, and an inference engine generating trajectories. The question is how you get new weights from the trainer into the inference engine without wrecking your throughput.

The current state in Ray is that you have to manage your own transport. If you want to push weights to an SGLang server, you serialize them, move them yourself, and call an update API. There's no first-class concept of "here is a reference to weights that live in the Ray object store, please load them." The gap between Ray's actor model and NCCL collectives has to be bridged by hand.

That's what the RFC is trying to fix.

## The Proposed API

The strawman is simple:

```python
weight_ref = ray.put(state_dict_shard)
ray.get(sglang_server.update_weights_from_object_ref.remote(weight_ref, version=42))
```

Instead of the caller managing how weights move, you put a shard into the Ray object store and pass the `ObjectRef` to the SGLang server. The server fetches it, loads it, and bumps its internal version counter. The caller never thinks about transport.

The flow inside the server on receipt of an ObjectRef:

1. **Pause** — stop accepting new requests, drain or hold in-flight ones
2. **Load** — fetch the ref from the object store, apply weights to GPU
3. **Post-process** — quantization, format conversion if needed
4. **Version bump** — increment the monotonic version counter
5. **Continue** — resume serving

Versioning is monotonic and strictly enforced: a server will not apply an update with a version lower than its current one. This means you can fire-and-forget weight updates from the trainer without worrying about out-of-order delivery corrupting state.

## Chunked Refs

For multi-billion parameter models, a single `ObjectRef` for the full state dict would be enormous — potentially tens of GBs. Putting that as one object has problems: it's a single point of pressure on the plasma store, it can't be pipelined, and it makes ref lifetime management blunt.

The design uses **chunked refs**: `List[ObjectRef]`, one per all-gather bucket. The server receives the list and processes chunks as they arrive. This gives us:

- **Pipelining**: the server can start processing chunk $i$ while chunk $i+1$ is still in flight
- **Finer-grained eviction**: the object store can evict chunks independently once consumed
- **Natural alignment with collective ops**: if the trainer is doing tensor-parallel training, the shards already exist as separate objects

The wrapper holds refs until the load is complete, then releases them. Whether to hold them defensively (in case of retry) or trust Ray's refcounter to handle lifetime is one of the open questions I'm still working through.

## Open Questions

The RFC is deliberate about what it doesn't decide yet. Here's where I'm still thinking:

**Transport layer.** Should this build on Ray Direct Transport (RDT) or use Mooncake as the primary path? RDT is Ray-native and would make ObjectRef-based transfer zero-copy within a cluster, but Mooncake has better RDMA support for cross-node transfers. The right answer probably depends on whether you're on a single node or spread across machines — but I'd rather not expose that distinction to the caller.

**Concurrent generation semantics.** When a weight update arrives while requests are in flight, you have three options: drain (wait for all in-flight to finish before applying), abort (kill in-flight requests and apply immediately), or inflight-continue (apply weights but let in-flight requests finish with the old weights). For training workloads, drain is usually correct — you want rollout batches to be generated under a consistent policy. But drain adds latency. The right default isn't obvious.

**Failure recovery.** If the load fails mid-way — OOM, node failure, whatever — do you silently pause the server (stop serving, wait for intervention), or attempt to roll back to the previous weights? Rolling back is safer from a correctness standpoint but requires keeping the previous state around, which costs memory. For now the RFC defers to silent pause, but I think this needs more thought for production use cases.

**Metadata beyond version.** Right now the API accepts a version integer and that's it. But routing decisions and sampling parameters often need to be coordinated with weight updates — if you're doing async sync, different replicas may be on different weight versions, and your scheduler should know that. Whether this metadata belongs in the weight update API or in a separate coordination layer is an open design question.

## What I've Learned

A few things I didn't fully appreciate before digging into this:

The object store is not free at model scale. Ray's plasma store is fast for small objects, but for weight tensors in the multi-GB range, you can hit eviction pressure quickly if you're not careful about when you release refs. The chunked design helps, but you still need to think about how many weight generations you keep live simultaneously.

Pause semantics are harder than they look. "Pause the inference engine" sounds simple, but in a distributed setting with multiple replicas, it means coordinating a pause across all of them simultaneously. If you pause some replicas while others are still serving, you get a window where the fleet is inconsistent — some replicas are running old weights, some are running new ones, and your scheduler may not know which is which. Getting this right requires a broadcast + barrier pattern, not just calling pause on each replica independently.

Monotonic versioning is load-bearing. It sounds like a minor implementation detail, but enforcing that servers never apply a lower-version update is what makes fire-and-forget weight pushes safe. Without it, a slow replica could apply a stale update after a more recent one, and you'd have no way to detect it from outside.

## Status

The RFC is open for community deliberation. The biggest decision before implementation is the transport question — RDT vs. Mooncake vs. supporting both with a fallback chain. If you have opinions on this or have run into similar problems building RL training infrastructure, I'd welcome the input on the [issue](https://github.com/ray-project/ray/issues/63120).
